## 1. プロジェクト概要

### 1.1 プロダクトコンセプト

> アーティストの音楽再生回数を伸ばすための、音楽ラジオ配信プラットフォーム

配信者がラジオ番組を配信し、番組内でApple Musicの楽曲を紹介・同時再生できるiOSアプリ。SoundCloud / Mixcloudに近いUXを持ちつつ、**ラジオ音声とApple Music楽曲の同時再生**で差別化する。

### 1.2 最重要機能: Apple Music楽曲の同時再生

iOSのデフォルト動作では1つのオーディオしか再生できないが、MusicKit for Swiftを使用し、自アプリの音声再生とApple Musicカタログ再生を共存させる。

```
┌──────────────────────────────────────┐
│          Music Radio App              │
│                                      │
│  [ラジオ音声] ─── AVAudioPlayer     │
│       +                              │
│  [Apple Music楽曲] ─── MusicKit     │
│       ↓                              │
│  同時再生（ミキシング）               │
│                                      │
│  ※Apple Musicサブスク未加入者:       │
│    30秒プレビュー再生にフォールバック  │
└──────────────────────────────────────┘
```

### 1.3 MVPスコープ判断

| 機能 | MVPスコープ | 理由 |
|---|---|---|
| 番組配信（録音済み音声アップロード） | ✅ 含む | コアフロー |
| Apple Music同時再生 | ✅ 含む | 最重要差別化機能 |
| お気に入り・フォロー | ✅ 含む | エンゲージメント基盤 |
| プロフィール・認証 | ✅ 含む | 必須 |
| 管理画面 | ✅ 含む | 運営に必須 |
| **ライブ配信** | **❌ 推奨: MVPから除外** | 下記参照 |

**ライブ配信を除外する推奨理由:**
1. MusicKit同時再生だけでも技術リスクが高い。ライブ配信を加えるとリスクが集中する
2. ライブ配信はメディアサーバー（Agora / AWS IVS等）が必要でインフラコストが跳ね上がる
3. 「ラジオ×Apple Music同時再生で再生回数が伸びるか」を検証するには録音済み番組で十分
4. データモデル・API設計にライブ配信の拡張ポイントを予め用意しておけば後から追加可能

## 3. 機能要件

### 3.1 TOP画面

- ヘッダー: ロゴ（TOP遷移）、プロフィール、サインイン/サインアウト
- お気に入り番組: 番組カード横スクロール + 一覧リンク
- フォロー中配信者: 配信者カード横スクロール + 一覧リンク
- おすすめ: 番組 + 配信者のレコメンド
- フッター: 運営会社、利用規約、プライバシーポリシー、お問い合わせ

### 3.2 番組画面 ★コア画面

**番組情報:** サムネイル、番組名、メッセージ、再生数、お気に入り数、配信日、配信者情報（サムネイル・名前・フォロワー数・フォローボタン）

**音声再生:** 波形表示（タップシーク）、再生時間表示、再生/停止

**Apple Music楽曲表示・再生 ★最重要:**
- 配信者が設定した楽曲をリスト表示（アートワーク、曲名、アーティスト名、再生ボタン）
- 楽曲タップでApple Musicをアプリ内再生（MusicKit for Swift）
- サブスク加入者: フル再生 / 未加入者: 30秒プレビュー
- ラジオ音声と楽曲のダッキング制御

**操作:** お気に入り追加/除外、フォロー、シェア（iOS ShareSheet）

**ミニプレイヤー（画面下常駐）:** 再生/停止、お気に入り、シェア、Apple Music楽曲再生。画面遷移中も再生維持。

### 3.3 配信者画面

壁紙、サムネイル、配信者名、フォロワー数、フォローボタン、メッセージ。番組一覧（30件ずつ無限スクロール）。

### 3.4 プロフィール画面

ニックネーム・サムネイル・メッセージの編集。配信中番組リンク + 番組配信導線。メニュー: お気に入り一覧、フォロー一覧、パスワード再設定、サインアウト。

### 3.5 番組配信機能

**音声アップロード:** mp3/m4a/wav対応、最大100MB、アップロード進捗表示

**番組編集:** 番組名、サムネイル、メッセージ、**楽曲設定（Apple Music URL + 再生タイミングHH:MM:SS）**、配信開始日時、配信開始/停止

**楽曲再生タイミング連動:**
```
ラジオ音声タイムライン:
|----[00:00]--------[05:30]--------[12:00]--------[30:00]---|
                       ↑               ↑
                    楽曲A表示        楽曲B表示
```

### 3.6 一覧画面

**お気に入り番組一覧 / 配信中番組一覧:** 番組カード（サムネイル、番組名、配信者名、再生/停止、再生数、再生時間、配信日、お気に入り、シェア）。30件ずつ無限スクロール。

**フォロー一覧:** 配信者カード（サムネイル、配信者名、フォロワー数、フォローボタン）。30件ずつ無限スクロール。

### 3.7 認証

| 画面 | 機能 |
|---|---|
| サインイン | メール+パスワード |
| サインアップ | メール入力 → メール確認 → 初期登録（ニックネーム+PW） |
| パスワード再設定 | メール送信 → 新パスワード入力 |

### 3.8 管理画面（Web）

ユーザー管理、番組管理、配信者管理、レポート・分析（再生数、ユーザー数、人気ランキング）、お問い合わせ管理。

## 4. MusicKit連携 — 技術要件

### 4.1 前提条件

- Apple Developer Program登録 + MusicKit API Key発行が必須
- ユーザーのApple Musicサブスクリプション: フル再生に必須。未加入者はプレビュー30秒のみ

### 4.2 実装方針

```swift
// ラジオ音声: AVAudioPlayer
// Apple Music: ApplicationMusicPlayer（MusicKit for Swift）

try AVAudioSession.sharedInstance().setCategory(
    .playback,
    options: [.mixWithOthers]  // 他のオーディオとミキシング許可
)

let player = ApplicationMusicPlayer.shared
// → AVAudioPlayerと共存可能
```

### 4.3 技術検証（PoC）が必要な項目

| # | 検証項目 | リスク | 代替案 |
|---|---|---|---|
| P-1 | AVAudioPlayer + ApplicationMusicPlayer の同時再生が実機で動作するか | **高** | ラジオ一時停止して楽曲のみ再生 |
| P-2 | .mixWithOthers での音量バランス制御 | 中 | duckingオプション、手動ボリューム |
| P-3 | サブスク未加入時の30秒プレビュー再生 | 中 | Apple Music導線リンクのみ |
| P-4 | バックグラウンド再生時の同時再生維持 | **高** | バックグラウンドではラジオのみ |
| P-5 | Apple Music URLからMusicKit Track IDへの変換 | 低 | CatalogSearch APIで解決可能 |
| P-6 | App Store審査でのMusicKit利用ポリシー準拠 | 中 | Guidelinesの事前確認 |

**推奨: Phase 1開始前に1-2週間のPoC期間を設け、P-1〜P-4を実機検証。**

## 6. 技術スタック

### 6.1 構成図

```
┌──────────────┐      ┌──────────────┐     ┌──────────────┐
│   iOS App     │      │  Backend API  │     │  Admin Web   │
│  Swift/SwiftUI│◄────►│  FastAPI or   │◄───►│  Next.js     │
│               │ REST │  Rails        │     │  shadcn/ui   │
│ - AVAudio     │      │               │     │              │
│ - MusicKit    │      │ - PostgreSQL  │     └──────────────┘
└──────────────┘      │ - S3 (音声)   │
                      └──────────────┘
```

### 6.2 技術選定

| レイヤー | 推奨 | 選定理由 |
|---|---|---|
| iOS | Swift + SwiftUI | MusicKit for SwiftはSwiftUI前提の設計。高速開発 |
| バックエンド | FastAPI (Python) or Rails | クライアントのスタック/運用体制に合わせて決定 |
| DB | PostgreSQL | Supabase or RDS |
| ストレージ | AWS S3 or Supabase Storage | 音声ファイル + 画像 |
| 管理画面 | Next.js + shadcn/ui | 独立Webアプリ |
| CI/CD | GitHub Actions + Fastlane | iOS配信自動化 |

## 7. データモデル

```sql
-- users: 認証ユーザー
-- user_profiles: ニックネーム、アバター、壁紙、メッセージ、フォロワー数
-- programs: 番組（title, audio_url, status, scheduled_at, play_count, program_type）
-- program_tracks: 番組内楽曲（apple_music_url, track_id, play_timing_seconds）
-- favorites: お気に入り（user_id × program_id）
-- follows: フォロー（follower_id × following_id）
-- program_plays: 再生ログ（program_id, user_id, duration_seconds）

-- ★将来のライブ配信対応: programs.program_type = 'recorded' | 'live'
-- ★将来のライブ配信対応: programs.live_stream_url
```

## 9. リスクと対策

| リスク | 影響度 | 対策 |
|---|---|---|
| MusicKit同時再生が技術的に不可能 | **致命的** | Phase 0 PoCで早期検証。不可なら外部リンク遷移に変更 |
| App Store審査でMusicKit利用が却下 | 高 | Apple Developer Guidelines事前確認 |
| 音声ストレージコスト増大 | 中 | ファイルサイズ制限、圧縮、CDN |
| バックグラウンド再生の不安定 | 中 | AVAudioSession最適化、実機検証 |
